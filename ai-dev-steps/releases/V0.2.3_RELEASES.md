# V0.2.3 发布说明 · Elasticsearch 客户端协议兼容性修复

## 背景

当前 SearchDBPad 的连接测试已经恢复，但在进入详情页或执行查询时，又遇到了新的协议兼容性错误：

```text
Content-Type header [application/vnd.elasticsearch+json; compatible-with=8] is not supported
```

这说明问题已经从“产品标识校验”转移到了“请求头内容类型”层面。

---

## 当前现象拆解

### 已解决的问题

1. `@elastic/elasticsearch` v8 的 product check 问题。
2. 通过自定义 `HttpConnection` 注入 `x-elastic-product: Elasticsearch` 后，客户端不再误判“不是 Elasticsearch”。

### 当前的新问题

`Client` 默认会发送 v8 兼容媒体类型：

- `application/vnd.elasticsearch+json; compatible-with=8`
- `application/vnd.elasticsearch+x-ndjson; compatible-with=8`
- `application/vnd.elasticsearch+json; compatible-with=8,text/plain`

而当前连接的服务端不接受这些 Content-Type，因此返回 406。

---

## 继续可能出现的其他问题

### 1. 不是单个接口的问题，而是所有主进程 ES 调用都会受影响

当前所有 ES 请求都通过 `buildEsClient()` 创建 client，涉及：

- `src/main/services/cluster.service.ts`
- `src/main/services/index.service.ts`
- `src/main/services/document.service.ts`
- `src/main/services/import.service.ts`
- `src/main/services/export.service.ts`

所以如果只修某一个接口，其他功能仍会继续报错。

### 2. 只改 product check 不够

当前已经通过响应头注入绕过了 product check，但请求头还是 v8 compatibility mode。

因此会出现：

- 连接测试成功
- 进入详情页失败
- 搜索、导入、导出、批量写入也失败

### 3. 未来还可能遇到“接口级”兼容差异

如果实际连接的不是标准 Elasticsearch，而是：

- OpenSearch
- 旧版本 Elasticsearch
- 被代理层改写过的 Elasticsearch

那么除了请求头之外，某些 API 的响应字段、路径行为、错误码语义也可能存在差异。

这类问题不是当前这次 406 的直接原因，但需要在验证阶段重点留意。

### 4. testConnection 与业务接口的实现方式不一致

`testConnection` 走的是裸 `fetch()`，而业务接口走 `@elastic/elasticsearch` client。

这会导致：

- 测试连接通过
- 业务页面仍失败

所以后续修复必须关注“统一的 client 构建逻辑”，不能只改测试连接。

---

## 推荐修复方向

## 总原则

把兼容性处理集中在 `buildEsClient()`，让所有主进程 ES 调用自动共享同一套协议配置。

---

## 方案结构

### 方案 A：保留 product check 绕过 + 去掉 v8 兼容 Content-Type

这是当前最直接、最小改动的方案。

#### 修改点

`src/main/services/esClient.ts`

#### 目标

1. 保留当前的 `CompatibleConnection extends HttpConnection`。
2. 继续注入 `x-elastic-product: Elasticsearch`，解决 product check。
3. 再额外覆盖 client 的请求头，改成标准媒体类型：
   - `application/json`
   - `application/x-ndjson`
   - `application/json, text/plain`

#### 预期效果

- 索引列表可加载
- 集群信息可加载
- 文档搜索可执行
- 导入 / 导出 / 批量写入也能继续走同一套 client

---

### 方案 B：如果后续发现服务端并非标准 Elasticsearch，再引入连接级兼容开关

只有在方案 A 仍然无法覆盖目标服务时，才考虑增加：

- `compatibilityMode: 'standard' | 'elastic8' | 'legacy'`
- 或在连接配置里记录目标产品类型

但这一步**不建议现在就做**，因为当前问题还可以先通过统一请求头解决，没必要提前扩展配置模型。

---

## 建议的实现顺序

### 第 1 步：统一 client 构建逻辑

确认所有 ES 调用都只通过 `buildEsClient()` 创建 client，不在其他文件里单独 new client。

### 第 2 步：修正请求头策略

在 `src/main/services/esClient.ts` 中，去掉 v8 compatibility headers，改用标准 JSON / NDJSON 头。

### 第 3 步：保留 product header 注入

继续使用自定义 `HttpConnection` 注入 `x-elastic-product`，避免 product check 再次报错。

### 第 4 步：跑全链路验证

重点验证：

- 连接测试
- 集群信息
- 索引列表
- 搜索
- 文档新增 / 更新 / 删除
- 导入
- 导出

---

## 需要修改的文件

### 只改核心文件

- `src/main/services/esClient.ts`

### 可能无需改动的文件

以下文件继续保持不变，只是会自动继承新的 client 配置：

- `src/main/services/cluster.service.ts`
- `src/main/services/index.service.ts`
- `src/main/services/document.service.ts`
- `src/main/services/import.service.ts`
- `src/main/services/export.service.ts`

## 验证清单

### 基础验证

1. `npm run typecheck`
2. `npm run dev`
3. 测试连接成功
4. 打开详情页无 406

### 功能验证

1. 能正常加载集群信息
2. 能正常加载索引列表
3. 能正常搜索文档
4. 能正常导入文档
5. 能正常导出文档

### 异常验证

1. 错误地址
2. 错误账号密码
3. 目标服务响应非 2xx

要求：错误能返回到前端，且不泄露密码。

---

## 风险提示

- 这次问题不是 UI 问题，也不是 IPC 问题，是主进程 ES client 的协议层问题。
- 如果后续发现服务端实际是 OpenSearch 或者有代理改写响应头，那么还需要继续分辨“兼容头”与“API 兼容性”两个层面。
- 这次修复应尽量集中在一个地方，避免把协议兼容逻辑散落到各个 service 里。

---

## 结论

当前最优先的修改方向是：

1. **保留** product check 绕过。
2. **去掉** `compatible-with=8` 请求头。
3. **统一** `buildEsClient()` 的协议配置。
4. **用全链路验证**确认 cluster / index / document / import / export 都正常。

如果这一步仍然失败，再考虑引入更细的连接模式开关。
