# V0.3.0 发布说明 · 搜索引擎适配层与 Elasticsearch 6.x-9.x 兼容重构

> 状态：✅ 已完成（2026-06-21）
> 目标版本：V0.3.0
> 发布日期：2026-06-21
> 范围：架构重构，不新增非 Elasticsearch 搜索引擎实现

## 1. 背景

当前 SearchDBPad 的搜索引擎访问链路以 Elasticsearch 为中心：

```text
renderer
  -> preload
    -> ipcMain
      -> main service
        -> @elastic/elasticsearch SDK
```

该结构在 V0.1.x / V0.2.x 阶段可以支撑 Elasticsearch MVP，但继续发展会遇到以下问题：

1. `@elastic/elasticsearch` 官方 SDK 与 Elasticsearch major 版本绑定较强。
2. SDK v8 对旧版本 Elasticsearch、OpenSearch、代理改写环境存在 product check 与 vendored Content-Type 兼容问题。
3. 未来如果支持 Solr / OpenSearch / Meilisearch / Typesense 等搜索引擎，继续沿用 `esClient.ts` 会导致代码强耦合。
4. 当前 service 层天然假设底层只有 Elasticsearch，未来容易出现大量 `if engineType === ...` 分支。
5. 用户可能只使用一种搜索引擎，希望打包时只包含对应适配器代码；也可能同时使用多种搜索引擎，需要全量打包。

因此 V0.3.0 需要从“ES SDK 客户端模式”升级为：

```text
搜索引擎 Adapter 架构 + Elasticsearch HTTP Adapter
```

---

## 2. 总体目标

### 2.1 当前必须达成

```text
1. 当前产品仍只支持 Elasticsearch。
2. Elasticsearch 支持 6.x / 7.x / 8.x / 9.x 四个大版本。
3. 不再依赖 @elastic/elasticsearch SDK 作为主请求链路。
4. 主进程通过统一 HTTP client 访问 Elasticsearch。
5. ES 版本差异全部收敛在 elasticsearch adapter 内部。
6. renderer / preload / IPC 安全边界不变。
7. 不引入 renderer 直接访问网络、文件系统或 shell 的能力。
```

### 2.2 架构必须预留

```text
1. 未来支持多个搜索引擎服务器。
2. 不同搜索引擎 adapter 低耦合。
3. 通用 service 不包含具体搜索引擎的实现细节。
4. 未来可以按搜索引擎裁剪打包。
5. 未来可同时打包多个 adapter。
```

### 2.3 当前不做

```text
1. 不实现 Solr。
2. 不实现 OpenSearch 独立 adapter。
3. 不实现 Meilisearch / Typesense。
4. 不开放多搜索引擎选择 UI。
5. 不做插件市场。
6. 不大规模重命名所有 UI 文案。
7. 不改变现有业务功能范围。
```

### 3.2 调用链路

目标链路：

```text
renderer
  -> preload
    -> ipcMain
      -> main service
        -> SearchEngineAdapter
          -> ElasticsearchAdapter
            -> HTTP client
```

service 层只负责：

```text
1. 根据 connectionId 读取连接。
2. 根据 connection.engineType 取得 adapter。
3. 调用 adapter 的通用方法。
4. 返回统一结果或统一错误。
```

service 层不负责：

```text
1. 拼接 Elasticsearch URL。
2. 判断 ES major 版本。
3. 处理 mapping type 差异。
4. 处理 Bulk NDJSON 细节。
5. 处理未来 Solr / OpenSearch 的特殊请求。
```

```ts
  createDocument(
    connection: SearchConnection,
    input: DocumentCreateInput
  ): Promise<DocumentWriteResult>

    input: ImportInput
  ): Promise<ImportResult>

    connection: SearchConnection,
    input: ExportInput
```

### 5.2 设计约束

1. adapter 接口使用产品能力命名，不使用 Elasticsearch 专属命名。
2. Elasticsearch 的 `_mapping` / `_settings` 当前可以先保留在通用接口中，因为 SearchDBPad 当前核心功能仍围绕 ES。
3. 未来如果某个搜索引擎不支持某能力，adapter 返回统一的 `SearchEngineError`，错误码为 `unsupported_capability`。
4. 通用 service 不允许直接 import `src/main/adapters/elasticsearch/*` 的具体 API 文件，只能通过 registry 获取 adapter。

## 6. Adapter Registry

### 6.1 新增 `src/main/search/adapterRegistry.ts`

当前只启用 Elasticsearch：

```ts
export async function getSearchEngineAdapter(
  type: SearchEngineType
): Promise<SearchEngineAdapter> {
  switch (type) {
    case 'elasticsearch': {
      const mod = await import('../adapters/elasticsearch')
      return mod.elasticsearchAdapter
    }
    default:
      throw new Error(`当前版本未启用搜索引擎适配器: ${type}`)
  }
}
```

### 6.2 禁止静态全量 import

禁止这样写：

```ts
import { elasticsearchAdapter } from '../adapters/elasticsearch'
import { solrAdapter } from '../adapters/solr'
```

原因：会导致所有 adapter 被打进产物，不利于后续裁剪打包。

请求头规则：

```text

Content-Type: application/json

```

```text

application/vnd.elasticsearch+x-ndjson; compatible-with=8
```

### 8.3 版本探测

`src/main/adapters/elasticsearch/detector.ts`

```ts
export async function detect(
  connection: SearchConnection
): Promise<SearchEngineServerInfo> {
  const root = await elasticsearchRequest<any>(connection, {
    method: 'GET',
    path: '/'
  })

  const version = String(root?.version?.number ?? '0.0.0')
  const [major = 0, minor = 0, patch = 0] = version
    .split('.')
    .map((part) => Number.parseInt(part, 10) || 0)

  return {
    engineType: 'elasticsearch',
    engineName: 'Elasticsearch',
    version,
    major,
    minor,
    patch,
    distribution: root?.version?.distribution ?? root?.version?.build_flavor,
    compatibleRange: '6.x-9.x'
  }
}
```

### 8.4 版本能力表

`src/main/adapters/elasticsearch/capabilities.ts`

```ts
export interface ElasticsearchCapabilities {
  supportsMappingTypes: boolean
  requiresDocTypeForMapping: boolean
  supportsTypelessDocumentApi: boolean
  supportsTypedDocumentApi: boolean
  supportsSeqNoPrimaryTerm: boolean
  supportsRuntimeFields: boolean
  supportsDataStreams: boolean
}

export function getElasticsearchCapabilities(info: SearchEngineServerInfo) {
  return {
    supportsMappingTypes: info.major <= 6,
    requiresDocTypeForMapping: info.major <= 6,
    supportsTypelessDocumentApi: info.major >= 7,
    supportsTypedDocumentApi: info.major <= 7,
    supportsSeqNoPrimaryTerm: info.major >= 7,
    supportsRuntimeFields: info.major >= 7,
    supportsDataStreams: info.major >= 7
  }
}
```

### 8.5 版本差异处理

`src/main/adapters/elasticsearch/versionCompat.ts`

#### 创建索引 Mapping

ES 6.x：

```json
{
  "mappings": {
    "_doc": {
      "properties": {
        "name": {
          "type": "text"
        }
      }
    }
  }
}
```

ES 7.x / 8.x / 9.x：

```json
{
  "mappings": {
    "properties": {
      "name": {
        "type": "text"
      }
    }
  }
}
```

适配函数：

```ts
export function normalizeCreateIndexBodyForEsVersion(
  body: unknown,
  info: SearchEngineServerInfo
): unknown {
  if (info.major <= 6) {
    // 若 body.mappings 已经包含 _doc，则保持原样
    // 否则包一层 _doc
  }
  return body
}
```

#### 文档 API

统一优先使用：

```http
POST /{index}/_doc
PUT /{index}/_doc/{id}
DELETE /{index}/_doc/{id}
```

该形式可覆盖 ES 6.x - 9.x 的主要场景。

#### Search API

统一使用：

```http
POST /{index}/_search
```

#### Bulk API

统一使用：

```http
POST /_bulk
Content-Type: application/x-ndjson
```

Bulk action 使用：

```json
{"index":{"_index":"users","_id":"1"}}
{"name":"张三"}
```

## 9. Service 层重构

### 9.1 新增 `src/main/services/searchEngine.service.ts`

```ts
export async function resolveAdapterByConnectionId(connectionId: string) {
  const connection = resolveConnection(connectionId)
  const adapter = await getSearchEngineAdapter(connection.engineType)
  return { connection, adapter }
}
```

### 9.2 cluster.service.ts

由：

```ts
const client = buildEsClient(conn)
return client.info()
```

改为：

```ts
const { connection, adapter } = await resolveAdapterByConnectionId(connectionId)
return adapter.getClusterInfo(connection)
```

### 9.3 index.service.ts

所有索引操作改为：

```ts
const { connection, adapter } = await resolveAdapterByConnectionId(connectionId)
return adapter.listIndices(connection)
```

或：

```ts
return adapter.createIndex(connection, input)
```

### 9.4 document.service.ts

所有文档操作改为：

```ts
return adapter.searchDocuments(connection, input)
return adapter.createDocument(connection, input)
return adapter.updateDocument(connection, input)
return adapter.deleteDocument(connection, input)
```

### 9.5 import/export service

导入导出格式解析仍可保留在 service 或迁移到 adapter。

V0.3.0 推荐：

```text
1. 文件选择、文件读取、CSV/JSON/NDJSON 解析逻辑保留在 import/export service。
2. 写入搜索引擎的逻辑交给 adapter.importDocuments / adapter.exportDocuments。
```

这样未来不同搜索引擎可以复用文件解析流程，但写入方式由 adapter 决定。

未来可新增：

```text

```

生成：

```ts
export const enabledEngines = ['elasticsearch'] as const

    case 'elasticsearch':
      return import('../adapters/elasticsearch')
  }
```

### 11.4 Electron Builder 文件过滤

未来如果多个 adapter 都被 tsc 编译到 `dist/main/adapters/`，需要通过 electron-builder `files` 过滤未启用 adapter：

```json

    "dist/main/**/*",
    "dist/preload/**/*",
    "dist/renderer/**/*",
    "dist/shared/**/*",
    "!dist/main/adapters/solr/**/*",
    "!dist/main/adapters/opensearch/**/*"
  ]
}
```

V0.3.0 不实现多 adapter 过滤，但目录设计必须支持。

## 14. 风险与注意事项

### 14.1 ES 6.x 风险

ES 6.x 的 mapping type 与 7.x+ 不同，创建索引时需要重点处理。

要求：

```text
所有 mapping type 兼容逻辑只能放在 elasticsearch/versionCompat.ts。
```

### 14.2 ES 9.x 风险

ES 9.x 对某些旧参数可能更严格。

要求：

```text
1. 默认请求体尽量保持最小化。
2. 不发送已废弃参数。
3. 错误返回要保留原始 details，便于定位。
```

### 14.3 多引擎风险

通用接口不能过度绑定 Elasticsearch。

要求：

```text
1. 新增接口时先判断是否是产品通用能力。
2. ES 专属能力优先放入 adapter 内部。
3. 如果确实需要暴露 ES 专属功能，命名要明确为 capability，而不是污染通用接口。
```

### 14.4 打包裁剪风险

V0.3.0 只实现 ES adapter，所以不会真正出现多 adapter 裁剪问题。

但代码必须避免静态全量 import，为后续 generated manifest 做准备。

---

## 15. 回滚方案

如果 V0.3.0 中途发现 HTTP adapter 改造风险过高：

```text
1. 保留当前 @elastic/elasticsearch SDK 方案。
2. 将新 adapter 基础设施合入但不启用。
3. 通过配置开关切换：legacy SDK / http adapter。
4. 等 HTTP adapter 验证完成后再删除 SDK。
```

但默认不建议长期保留双链路，避免后续维护成本翻倍。

## 16. 完成标准

V0.3.0 完成时必须满足：

```text
1. 主流程不再依赖 @elastic/elasticsearch 发请求。

3. 业务 service 不包含 ES 版本差异判断。
4. ES 版本兼容逻辑集中在 elasticsearch adapter。
5. SearchEngineAdapter 接口稳定。
6. 旧连接数据兼容。
7. IPC 安全边界不变。
8. 打包仍可生成 Windows exe。
9. releases/V0.3.0_RELEASES.md 与实际实现保持同步。
```
